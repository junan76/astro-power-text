Markdown 文件适合长期维护技术文章。它不依赖特定后台，也容易放进 Git 里管理历史。

我的理想流程是：

1. 在本地创建 Markdown 文件。
2. 先写清楚标题、摘要和标签。
3. 完成正文后运行构建命令。
4. 检查页面、RSS 和标签页是否正常。
5. 提交代码并部署。

这个流程的关键是让内容和代码使用同一套版本管理机制。文章不是数据库里的记录，而是仓库里可以被审查、比较和回滚的文件。
