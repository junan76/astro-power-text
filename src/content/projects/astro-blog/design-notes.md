这是一份事后的设计备忘。把"为什么"写下来，是为了几个月后再回头看时不需要重新推一遍。

## 为什么分 posts / projects 两个集合

一开始尝试过只用一个 `posts` 集合，靠 tag 区分专题。结果发现两类内容的需求其实不同：

- 普通文章是"独立的小篇"，列表里按时间倒序就好。
- 围绕同一个主题写的多篇笔记需要一个共同入口，并且这个入口本身不应该出现在"最近文章"里抢位置。

把项目独立成一个集合后，过滤逻辑变得很自然：项目主页通过 `isProjectIndex` 自动从归档和 RSS 中剔除，子文章则混入时间线。

## 为什么 URL 直接用目录路径

很多博客会把目录映射成"分类"，再单独生成分类页。这里没有走那条路，理由是：

- 分类页和标签页功能高度重合，再多一套维护代价。
- 直接把目录暴露在 URL 里，迁移和重命名时心智负担更小——文件路径和 URL 一一对应。

代价是：如果想给某一类文章加聚合页，目前只能通过 tag。这是一个有意识的取舍。

## 为什么 hover 时浮层显示描述

主页和归档的列表是密集的"日期 + 标题"两栏。直接把描述塞进去会让列表变胖，破坏一目十行的浏览节奏。但描述对判断"要不要点进去"又很有用。最终方案是把描述挂在 `data-description` 属性上，hover 时通过 CSS 浮层展示——常态紧凑，需要时可见。
